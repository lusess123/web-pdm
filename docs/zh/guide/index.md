# 清晰呈现复杂的数据关系

`web-pdm` 是一个轻量、可嵌入的 React ER 图组件，也是一项持续建设“在线版 PowerDesigner”的开源实践。它把模型元数据转换成交互式 Canvas，并提供模型导航、字段关系、自动布局、缩放、小地图和图片导出等能力。

![web-pdm ER 图](../../assets/erd.jpeg)

## 什么是 ER 图？

E-R 方法是“实体—联系方法”（Entity–Relationship Approach）的简称，是描述现实世界概念结构模型的有效方法；ER 图则是这种方法在界面上的可视化呈现。

无论是常见的数据库驱动开发、强调领域边界的 DDD、模型驱动的 Low-code 平台，还是由主数据驱动的 APaaS 平台，ER 图都提供了一种观察和讨论业务的共同视角。

## 为什么需要 ER 工作台？

### 业务设计 ≈ 模型设计 ≈ 数据库设计

一个成熟的 APaaS 或 Low-code 平台，需要隔离技术复杂度与业务复杂度。模型元数据是系统的重要输入，不同项目拥有不同的模型与字段，因此 ER 图也可以从模型数量、关系和边界上度量系统的业务复杂度。

在业务复杂的大型项目中，团队通常借助 DDD 进行业务设计，领域模型是其中非常重要的一部分。模型设计者往往是领域专家，而功能实现者是工程团队；一张可视化 ER 图可以作为双方之间的重要中间产物。

在中小型项目中，由于业务和界面交付相对直接，视图模型（VO）、业务模型（BO）和数据模型（DO）往往趋向三者合一，此时 ER 图也承担数据库设计工具的角色。

在项目 Kick-off 或方案评审中，团队经常会展开 ER 图共同讨论。对许多中小型项目而言，ER 图一旦确定，核心业务逻辑和工作量也随之清晰。这就是工程师常说的“在项目编码前严格把控数据库设计”。

### 定制化的 ER 图更有价值

任何真正落地的 APaaS 或 Low-code 平台都很难完全通用。平台建设通常不是从零开始，而是对公司既有技术栈、项目交付过程和业务经验的抽象，因此必然带有特定业务领域的特点。技术栈与业务属性的结合，恰恰形成了平台的价值和竞争力。

每个组织都会形成一些约定俗成的业务概念和资源，以减少沟通成本。模型是这类平台的中心，ER 图则是呈现这些概念的最佳舞台。通过对通用 ERD 进行业务化扩展，领域专家可以直接在图上表达和创造业务。

### 在线版本的 PowerDesigner

PowerDesigner 是经典的 ER 图设计工具。除了基础能力扎实，它对中文元信息的支持也影响了许多国内数据库设计流程；而不少同类工具由海外公司开发，对中文语义的支持相对有限。

传统 PowerDesigner 主要面向 Windows，macOS 和跨平台协作场景长期缺少同等顺手的替代品。因此，除了产品自身的嵌入需求，做一个能够在浏览器中运行、跨平台分享的在线 ER 图工具，也是 web-pdm 的重要缘起。

## 如何实现？

### Canvas 与 SVG 的选择

下面保留项目最初技术选型时使用的对比结论：

| Canvas                                   | SVG                               |
| ---------------------------------------- | --------------------------------- |
| 依赖分辨率                               | 不依赖分辨率                      |
| 原生事件处理能力较弱，需要由上层引擎封装 | 每个图形元素都可以绑定事件        |
| 文本与可访问性能力需要额外处理           | 文本和 DOM 语义更自然             |
| 可以导出为 PNG 或 JPEG                   | 复杂度升高时，大量 DOM 会拖慢渲染 |
| 适合大量对象频繁重绘的图像密集场景       | 更适合对象数量可控的矢量界面      |

相比 SVG，Canvas 更接近底层绘图能力，也是通往 WebGL 的入口，留有更大的性能优化空间。对标 PowerDesigner 的 Web ER 图，核心目标之一是展示成百上千个模型，因此项目从一开始就选择 Canvas。

Canvas 的绘图 API 较底层，直接开发会影响效率；G6 提供了图数据、布局、交互与渲染抽象，弥补了这部分差距。当前版本已经升级到 G6 5，并通过独立运行时适配层隔离异步生命周期、自定义节点、具名端口和布局实现。

### 为嵌入业务系统而设计

web-pdm 是一个组件，而不是托管式建模服务。模型数据、事件回调、国际化文案和外围界面始终由接入应用掌控。你可以使用默认的轻量控件，也可以通过组件适配接口替换它们。

公开的模型与事件接口尽量保持向后兼容；G6 本身则被封装在组件内部，使宿主应用不需要随着 G6 的大版本变化而重写业务代码。
